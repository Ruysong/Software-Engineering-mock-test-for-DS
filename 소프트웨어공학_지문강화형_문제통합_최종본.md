목적: 설명강화 지문을 먼저 읽고, 바로 이어서 같은 번호의 개념 문제와 정답/해설까지 확인할 수 있도록 합친 최종 공부 파일

## 1. 설계의 필요성과 전체 흐름

### 강화 지문

소프트웨어 개발은 단순히 코드를 많이 쓰는 일이 아니다. 프로그램이 작을 때는 머릿속으로 전체 구조를 떠올리며 바로 만들 수 있지만, 규모가 커지면 이야기가 달라진다. 기능이 많아지고 사람이 여러 명 붙고 수정 요청이 계속 들어오면, 아무 구조 없이 작성된 코드는 금방 얽힌다. 처음에는 “일단 돌아가니까 됐다”고 생각할 수 있지만, 나중에 기능 하나를 고치려 할 때 예상하지 못한 곳이 함께 망가지면 유지보수가 매우 어려워진다.

그래서 요구사항과 구현 사이에 설계가 필요하다. 요구사항 분석이 사용자가 원하는 기능과 조건을 정리하는 단계라면, 설계는 그 요구사항을 어떤 구조로 실현할지 정하는 단계이다. 즉 요구사항 분석이 “무엇을 만들 것인가”에 가깝다면, 설계는 “어떻게 나누고 연결해서 만들 것인가”에 가깝다. 이 단계에서 시스템을 기능 단위로 나누고, 각 단위의 책임을 정하며, 단위들이 어떻게 협력할지를 결정한다.

이 흐름을 잡아 두면 시험 범위 전체가 훨씬 덜 흩어진다. 설계는 전체적인 구조 잡기이고, 아키텍처는 그중에서도 시스템의 큰 골격을 정하는 일이다. 클래스 설계는 객체들이 어떤 관계를 맺는지 정하는 일이고, 구현은 설계를 실제 코드로 옮기는 일이다. 디자인 패턴은 자주 반복되는 설계 문제를 해결하기 위한 검증된 방법이며, 테스트는 만든 결과가 요구사항을 만족하는지 확인하는 활동이다.

따라서 이 과목의 핵심 질문은 “소프트웨어를 어떻게 덜 복잡하고, 덜 위험하고, 더 고치기 쉽게 만들 것인가”이다. 설계와 아키텍처는 복잡도를 줄이고, 구현 규칙은 코드를 읽기 쉽게 하며, 패턴은 반복되는 문제의 해결책을 제공하고, 테스트는 오류를 찾아 신뢰를 높인다. 각각의 단원은 따로 외울 것이 아니라 개발 과정 속에서 어디에 놓이는지 함께 봐야 한다.

확인 포인트:
- 요구사항 분석은 “무엇을”, 설계는 “어떻게 구조화할지”에 가깝다.
- 구현은 설계 결과를 실제 코드로 옮기는 단계이다.
- 테스트는 만든 소프트웨어가 요구사항을 만족하는지 확인한다.

1. ④. 구현은 설계 이후에 설계를 코드로 옮기는 단계이다.
2. ②. 설계는 시스템을 나누고 각 단위의 책임과 협력 방식을 정한다.
3. ③. 기본 흐름은 요구사항 분석, 설계, 구현, 테스트이다.
4. ①. 디자인 패턴은 반복되는 설계 문제의 재사용 가능한 해결책이다.
5. ①. 설계가 부족하면 기능이 얽히고 수정 영향 범위를 알기 어렵다.

## 2. 분할과 정복, 추상화

### 강화 지문

복잡한 문제를 해결할 때 가장 먼저 필요한 생각은 “한 번에 다 하려 하지 않는 것”이다. 큰 문제를 작은 문제로 나누고, 각각을 해결한 뒤 다시 전체로 연결하는 방식이 분할과 정복이다. 소프트웨어 설계에서도 이 사고가 중요하다. 하나의 거대한 프로그램을 그대로 다루면 어떤 기능이 어디에 있는지 알기 어렵고, 오류가 발생해도 원인을 찾기 어렵다. 그래서 시스템을 여러 기능 단위로 나누어 생각한다.

다만 분할과 정복은 무조건 잘게 쪼개라는 뜻이 아니다. 너무 많이 나누면 조각이 많아져 오히려 연결 관계가 복잡해질 수 있다. 중요한 것은 “무엇을 기준으로 나눌 것인가”이다. 관련 있는 기능은 같은 단위 안에 모으고, 관련 없는 기능은 분리해야 한다. 이 기준이 뒤에서 나오는 응집도와 결합도와 연결된다. 내부 기능이 한 목적에 잘 모이면 응집도가 높고, 모듈끼리 덜 얽혀 있으면 결합도가 낮다.

추상화는 분할과 정복과 함께 복잡도를 낮추는 중요한 설계 원리이다. 추상화는 세부 사항을 모두 버린다는 뜻이 아니라, 지금 이해에 필요한 핵심만 남기고 나머지는 감추는 것이다. 예를 들어 “결제 처리”라는 기능을 볼 때, 처음부터 카드 승인, 포인트 차감, 영수증 발급, 예외 처리의 모든 코드를 동시에 생각하면 복잡하다. 먼저 결제 처리라는 큰 기능으로 이해하고, 세부 절차는 나중에 다루면 된다.

추상화를 잘하면 설계자는 큰 구조를 먼저 볼 수 있다. 사용자는 자동차 엔진의 내부 원리를 몰라도 운전할 수 있고, 개발자는 어떤 모듈의 내부 구현을 몰라도 정해진 인터페이스를 통해 사용할 수 있다. 이처럼 추상화는 불필요한 세부를 감추고 중요한 관계를 드러내어, 설계자가 복잡한 시스템을 다룰 수 있게 해 준다.

확인 포인트:
- 분할과 정복은 큰 문제를 작은 문제로 나누어 해결하는 사고이다.
- 잘 나누는 기준은 높은 응집도와 낮은 결합도와 연결된다.
- 추상화는 핵심만 남기고 불필요한 세부 사항을 감추는 것이다.

## 3. 캡슐화, 정보 은닉, 다형성

### 강화 지문

캡슐화는 관련 있는 데이터와 기능을 하나의 단위로 묶는 것이다. 객체지향에서 클래스가 대표적인 예이다. 예를 들어 계좌를 생각해 보자. 계좌에는 잔액이라는 데이터가 있고, 입금과 출금이라는 기능이 있다. 이 데이터와 기능을 여기저기 흩어 놓으면 계좌라는 개념을 이해하기 어렵다. 반대로 계좌와 관련된 데이터와 기능을 하나의 클래스 안에 묶으면 구조가 분명해진다.

정보 은닉은 캡슐화와 자주 함께 등장하지만 초점이 다르다. 캡슐화가 “묶는 것”이라면, 정보 은닉은 “외부가 몰라도 되는 내부 구현을 숨기는 것”이다. 계좌 잔액을 외부 코드가 직접 마음대로 바꾸게 하면 잘못된 값이 들어갈 수 있다. 대신 입금과 출금 같은 정해진 기능을 통해서만 잔액이 바뀌게 하면 데이터가 보호된다. 내부 구현 방식이 바뀌어도 외부는 같은 방식으로 사용할 수 있으므로 변경의 영향도 줄어든다.

다형성은 같은 이름의 동작이 상황에 따라 다르게 실행될 수 있는 성질이다. 여러 도형이 모두 `draw()`라는 기능을 가진다고 해 보자. 외부 코드는 원인지 사각형인지 세부 종류를 일일이 판단하지 않고 “그려라”라고만 요청할 수 있다. 그러면 원은 원대로, 사각형은 사각형대로 자기 방식으로 그린다. 이런 구조는 구체적인 종류가 늘어나도 외부 코드의 부담을 줄인다.

다형성과 함께 중복정의와 재정의도 구분해야 한다. 중복정의는 같은 이름의 메서드를 매개변수 차이로 여러 개 두는 것이다. 예를 들어 `print(int)`와 `print(String)`은 이름은 같지만 입력이 다르다. 재정의는 상속 관계에서 상위 클래스의 메서드를 하위 클래스가 다시 구현하는 것이다. 시험에서는 중복정의와 재정의를 자주 바꾸어 묻기 때문에, “매개변수 차이”와 “상속 관계에서 다시 구현”이라는 기준을 기억해야 한다.

확인 포인트:
- 캡슐화는 관련 데이터와 기능을 묶는 것이다.
- 정보 은닉은 내부 구현을 숨겨 외부 영향과 오류 가능성을 줄인다.
- 중복정의는 매개변수 차이, 재정의는 상속 관계에서 다시 구현이다.

   4. 객체를 반드시 하나만 생성한다.
   5. 테스트 케이스를 의사결정 테이블로 바꾼다.

2. 정보 은닉의 목적에 가장 가까운 것은?
   1. 외부에서 내부 데이터를 아무렇게나 수정하게 하는 것
   2. 내부 구현을 숨겨 변경의 영향을 줄이는 것
   3. 모든 클래스를 하나의 파일에 작성하는 것
   4. 모듈 간 의존성을 높이는 것
   5. 조건문의 모든 경로를 실행하는 것

3. 중복정의와 재정의의 구분으로 적절한 것은?
   1. 중복정의는 상속 관계에서만 가능하고, 재정의는 매개변수 차이로 구분된다.
   2. 중복정의는 매개변수 차이로 구분되고, 재정의는 상속 관계에서 다시 구현하는 것이다.
   3. 둘 다 테스트 절차의 한 단계이다.
   4. 둘 다 아키텍처 스타일의 종류이다.
   5. 중복정의는 객체 생성 패턴이고, 재정의는 구조 패턴이다.

4. 다음 중 다형성의 설명으로 가장 적절한 것은?
   1. 같은 호출이 객체에 따라 다르게 동작할 수 있다.
   2. 모든 객체가 같은 내부 데이터를 공유한다.
   3. 프로그램 실행 없이 오류를 찾는다.
   4. 입력 값을 같은 영역으로 나눈다.
   5. 시스템의 물리적 배치를 결정한다.

5. 윗글의 내용과 일치하지 않는 것은?
   1. 캡슐화는 관련 데이터와 기능을 묶는다.
   2. 정보 은닉은 외부가 알 필요 없는 내부 구현을 숨긴다.
   3. 캡슐화와 정보 은닉은 초점이 완전히 동일하다.
   4. 재정의는 상속 관계와 관련된다.
   5. 다형성은 같은 이름의 동작이 다르게 실행될 수 있음을 의미한다.

### 정답 및 해설

1. ①. 캡슐화는 관련 데이터와 기능을 하나로 묶는다.
2. ②. 정보 은닉은 내부 구현을 숨겨 외부 영향과 오류 가능성을 줄인다.
3. ②. 중복정의는 매개변수 차이, 재정의는 상속 관계에서 다시 구현이다.
4. ①. 다형성은 같은 메시지가 객체에 따라 다르게 동작하는 성질이다.
5. ③. 둘은 함께 쓰이지만 캡슐화는 묶기, 정보 은닉은 숨기기에 초점이 있다.

## 4. 모듈화, 응집도, 결합도, UI 설계

### 강화 지문

모듈화는 시스템을 작은 기능 단위인 모듈로 나누는 것이다. 그런데 여기서 중요한 점은 “나누었다”는 사실 자체가 아니라 “어떻게 나누었는가”이다. 프로그램을 여러 파일이나 여러 함수로 나누었다고 해서 자동으로 좋은 구조가 되는 것은 아니다. 한 모듈 안에 로그인, 결제, 통계, 알림처럼 서로 관련 없는 기능이 뒤섞여 있다면 그 모듈은 이름만 모듈일 뿐 실제로는 이해하기 어려운 덩어리이다.

좋은 모듈을 판단할 때 쓰는 기준이 응집도와 결합도이다. 응집도는 모듈 내부의 기능들이 얼마나 하나의 목적에 잘 모여 있는지를 나타낸다. 예를 들어 “회원 인증 모듈” 안에 로그인, 로그아웃, 비밀번호 확인처럼 인증과 관련된 기능이 모여 있다면 응집도가 높다고 볼 수 있다. 반대로 인증 모듈 안에 결제 정산이나 게시판 통계 기능이 함께 들어 있다면 응집도가 낮다.

결합도는 모듈과 모듈 사이의 의존 정도를 뜻한다. 한 모듈을 고치려면 다른 모듈 여러 개를 함께 고쳐야 한다면 결합도가 높은 구조이다. 특히 다른 모듈의 내부 구현을 직접 알고 있어야만 동작한다면 변경에 매우 약해진다. 좋은 설계는 내부적으로는 관련 기능이 잘 모인 높은 응집도를 원하고, 외부적으로는 불필요한 의존이 적은 낮은 결합도를 원한다. 시험에서는 “응집도는 내부, 결합도는 외부”라는 구분이 아주 중요하다.

사용자 인터페이스 설계는 사용자가 시스템을 어떻게 보고 조작할지를 정하는 설계이다. 기능이 아무리 많아도 사용자가 찾기 어렵고 실수하기 쉬우면 좋은 소프트웨어라고 보기 어렵다. UI 설계는 예쁜 화면만 만드는 일이 아니라 사용자가 자신의 목표를 쉽게 달성하도록 돕는 일이다. 같은 기능은 비슷한 위치와 방식으로 제공되어야 하고, 오류가 발생했을 때 사용자가 무엇을 해야 하는지도 분명히 안내해야 한다. 즉 UI 설계는 내부 알고리즘보다 사용자와 시스템의 만나는 지점을 다룬다.

확인 포인트:
- 모듈화는 시스템을 기능 단위로 나누는 것이다.
- 응집도는 모듈 내부 관련성이고 높을수록 좋다.
- 결합도는 모듈 간 의존성이고 낮을수록 좋다.

4. ④. 내부 분기 경로 검사는 화이트박스 테스트에 가깝다.
5. ④. UI 설계는 사용자의 이해와 조작을 돕는다.

## 5. 아키텍처와 품질 속성

### 강화 지문

아키텍처는 소프트웨어의 큰 골격이다. 작은 프로그램에서는 클래스나 함수 몇 개만으로도 전체 구조를 이해할 수 있지만, 대형 시스템에서는 전체를 먼저 보지 않으면 부분들이 제멋대로 만들어질 수 있다. 아키텍처 설계는 시스템을 이루는 주요 구성 요소가 무엇인지, 그 구성 요소들이 어떻게 연결되는지, 어떤 인터페이스를 통해 상호작용하는지를 정한다.

아키텍처는 단순히 구조도를 그리는 작업이 아니다. 시스템의 품질에도 직접 영향을 준다. 예를 들어 많은 사용자가 동시에 접속하는 서비스라면 성능이 중요하고, 금융 시스템이라면 보안성과 가용성이 중요하다. 가용성은 시스템이 장애 없이 계속 서비스를 제공할 수 있는 능력이다. 성능은 요청이 들어왔을 때 얼마나 빠르고 효율적으로 처리하는지를 뜻한다. 보안성은 허용되지 않은 접근을 막고 올바른 사용자에게만 서비스를 제공하는 능력이다. 사용성은 사용자가 시스템을 쉽게 이해하고 사용할 수 있는 정도이다.

품질 속성은 서로 충돌할 수 있다. 보안을 강화하면 로그인 절차가 복잡해져 사용성이 떨어질 수 있고, 성능을 높이기 위해 비용이 증가할 수도 있다. 그래서 아키텍처 설계에서는 어떤 품질 속성이 더 중요한지 우선순위를 정해야 한다. 모든 품질을 동시에 최고 수준으로 만족시키는 것은 현실적으로 어렵기 때문이다.

비즈니스 품질 속성도 아키텍처와 관련된다. 시장 적시성은 적절한 시기에 제품을 출시하는 능력과 관련된다. 아무리 잘 만든 소프트웨어라도 출시 시기를 놓치면 경쟁력을 잃을 수 있다. 예상 시스템 수명도 중요하다. 오래 사용할 시스템이라면 확장성, 유지보수성, 이식성을 더 신경 써야 한다. 개념적 무결성은 전체 시스템의 설계와 세부 구성 요소의 설계가 일관성을 유지해야 한다는 뜻이다.

확인 포인트:
- 아키텍처는 시스템의 큰 골격과 구성 요소 관계를 정한다.
- 품질 속성에는 가용성, 성능, 보안성, 사용성 등이 있다.
- 개념적 무결성은 전체와 부분 설계의 일관성과 관련된다.

---

### 문제

1. 아키텍처에 대한 설명으로 적절한 것은?
   1. 코드의 세부 문법만 정하는 작업이다.
   2. 시스템의 큰 골격과 구성 요소 관계를 정한다.
   3. 완성된 프로그램을 사용자에게 인수시키는 테스트이다.

   5. 입력 값을 동등한 영역으로 나누는 테스트이다.

   5. 시스템을 다른 환경으로 옮기는 능력

   1. 부당한 접근을 차단하고 올바른 사용자에게 서비스를 제공한다.
   2. 한 모듈 내부의 기능이 하나로 모이는 정도이다.
   3. 프로그램의 모든 문장을 실행하는 테스트 기준이다.
   4. 객체를 하나만 생성하도록 보장한다.
   5. 상위 클래스와 하위 클래스의 관계이다.

4. 다음 중 비즈니스 품질 속성에 가까운 것은?
   1. 시장 적시성
   2. 문장 검증 기준
   3. 중복정의
   4. 경계 값 분석
   5. 동료 검토

   3. 객체를 동적으로 장식하는 것이다.
   4. 내부 구현을 모두 외부에 공개하는 것이다.
   5. 프로그램을 실행하지 않고만 테스트하는 것이다.

### 정답 및 해설

1. ②. 아키텍처는 시스템의 큰 구조와 구성 요소 간 관계를 다룬다.
2. ③. 가용성은 장애 없이 서비스를 제공할 수 있는 능력이다.
3. ①. 보안성은 허용되지 않은 접근을 막는 능력이다.
4. ①. 시장 적시성은 비즈니스 품질 속성에 해당한다.
5. ②. 개념적 무결성은 전체와 부분 설계의 일관성과 관련된다.

## 6. 4+1 관점과 아키텍처 스타일

### 강화 지문

하나의 시스템도 보는 사람에 따라 관심사가 다르다. 사용자는 “이 시스템이 어떤 기능을 제공하는가”에 관심이 있고, 개발자는 “내부 클래스와 모듈이 어떻게 구성되는가”에 관심이 있다. 운영 담당자는 “어떤 서버에 어떻게 배치되는가”에 관심이 있을 수 있다. 그래서 아키텍처를 하나의 관점으로만 설명하면 부족하다. 4+1 관점은 이런 여러 관심사를 나누어 보는 방법이다.

`+1`에 해당하는 시나리오 또는 유스케이스 관점은 사용자가 인식하는 기능을 중심으로 본다. 논리적 관점은 시스템 내부의 클래스, 객체, 컴포넌트와 관계를 본다. 개발 관점은 소스 코드, 모듈, 실행 파일처럼 구현 단위가 어떻게 구성되는지를 본다. 프로세스 관점은 실행 중 동시성이나 성능 같은 문제에 관심을 둔다. 물리적 관점은 실제 하드웨어와 네트워크 환경에 시스템이 어떻게 배치되는지를 본다.

아키텍처 스타일은 반복적으로 사용되는 시스템 구조의 전형적인 형태이다. 매번 처음부터 구조를 새로 고민하기보다, 이미 많이 사용되고 검증된 스타일을 참고하면 시행착오를 줄일 수 있다. 개발자들끼리 같은 스타일 이름을 공유하면 의사소통도 쉬워진다. 예를 들어 “이 시스템은 MVC 구조로 가자”라고 하면 모델, 뷰, 제어를 나누자는 큰 방향을 빠르게 공유할 수 있다.

다만 아키텍처 스타일은 무조건 외워서 아무 시스템에나 적용하는 것이 아니다. 데이터 공유가 핵심인 시스템에는 데이터 중심형이 어울릴 수 있고, 네트워크를 통해 서비스를 요청하는 구조에는 클라이언트-서버 스타일이 어울릴 수 있다. 역할을 명확히 나누고 싶다면 계층 스타일이 적합할 수 있다. 결국 스타일은 시스템의 성격과 품질 요구에 맞게 선택해야 한다.

확인 포인트:
- 4+1 관점은 이해관계자의 관심사를 나누어 시스템을 본다.
- 유스케이스 관점은 사용자 기능, 논리적 관점은 내부 구조에 초점을 둔다.
- 아키텍처 스타일은 검증된 구조적 해결 방식이다.

## 7. 주요 아키텍처 스타일

### 강화 지문

데이터 중심형 스타일은 주요 데이터가 중앙 리포지토리에서 관리되는 구조이다. 여러 서브시스템은 이 중앙 저장소에 접근하여 데이터를 읽거나 쓴다. 핵심은 “공유 데이터가 중심에 있다”는 점이다. 은행 업무 시스템처럼 여러 기능이 같은 고객 정보나 거래 정보를 사용해야 하는 경우에 적합하다. 장점은 데이터 일관성을 관리하기 쉽다는 것이지만, 중앙 저장소가 병목이나 장애 지점이 될 수 있다는 점도 생각해야 한다.

클라이언트-서버 스타일은 클라이언트가 서버에 서비스를 요청하고 서버가 처리 결과를 돌려주는 구조이다. 웹 서비스에서 브라우저가 클라이언트이고 웹 서버가 서버인 경우를 떠올리면 쉽다. 씬 클라이언트는 클라이언트가 주로 화면 표시만 담당하고, 데이터 관리나 주요 처리는 서버가 맡는 형태이다. 서버 중심으로 관리할 수 있다는 장점이 있지만, 서버에 부담이 집중될 수 있다.

계층 스타일은 기능을 여러 계층으로 나누어 역할을 분리한다. 3계층 구조에서는 프레젠테이션 계층이 화면과 사용자 인터페이스를 담당하고, 비즈니스 로직 계층이 실제 기능 처리와 규칙을 담당하며, 데이터 계층이 데이터 저장과 관리를 담당한다. 역할이 분리되어 있으면 한 계층을 수정해도 다른 계층에 미치는 영향을 줄일 수 있다.

MVC 스타일은 모델, 뷰, 제어를 분리한다. 모델은 데이터와 핵심 상태를 담당하고, 뷰는 사용자에게 보이는 화면을 담당하며, 제어는 사용자의 입력을 받아 모델과 뷰를 연결한다. 데이터 흐름 스타일은 데이터를 여러 처리 단계가 차례로 변환하여 다음 단계로 넘기는 구조이다. 데이터 중심형이 중앙 저장소를 강조한다면, 데이터 흐름 스타일은 처리 과정의 흐름을 강조한다.

확인 포인트:
- 데이터 중심형은 중앙 리포지토리가 핵심이다.
- 클라이언트-서버는 요청과 응답 관계가 핵심이다.
- 계층, MVC, 데이터 흐름은 각각 역할 분리, 모델-뷰-제어 분리, 단계적 데이터 변환에 초점을 둔다.

4. ①. MVC는 Model, View, Controller이다.
5. ②. 데이터 흐름 스타일은 처리 단계별 데이터 변환이 핵심이다.

## 8. 클래스 간 관계

### 강화 지문

클래스 설계에서는 클래스들이 서로 어떤 관계를 맺는지 표현해야 한다. 연관 관계는 두 클래스의 객체가 의미 있는 연결을 가지는 관계이다. 교수와 학생이 수업이라는 관계로 연결되는 경우를 생각하면 쉽다. 단방향 연관은 한쪽만 상대를 아는 관계이고, 양방향 연관은 서로를 아는 관계이다. 다중성은 관계에 참여할 수 있는 객체 수를 나타낸다. 예를 들어 교수 한 명이 여러 학생을 가르치면 1:n 관계가 될 수 있다.

일반화 관계는 여러 클래스의 공통점을 뽑아 상위 클래스로 표현하는 관계이다. 객체지향의 상속과 연결되며 `is-a` 관계로 이해하면 쉽다. “학생은 사람이다”, “교수는 사람이다”처럼 말이 자연스러우면 학생과 교수는 사람의 하위 클래스로 볼 수 있다. 일반화는 공통 속성과 기능을 상위 클래스로 올려 중복을 줄이고 다형성을 활용할 수 있게 한다.

집합 관계와 합성 관계는 모두 전체와 부분의 관계이지만 강도가 다르다. 집합 관계는 부분 객체가 전체와 독립적으로 존재할 수 있는 비교적 약한 관계이다. 컴퓨터와 모니터를 생각하면, 모니터는 컴퓨터의 일부로 볼 수 있지만 컴퓨터와 분리되어도 존재할 수 있다. 합성 관계는 더 강하다. 부분 객체가 전체 객체에 강하게 종속되고 생명주기를 함께한다. 전체가 사라지면 부분도 함께 사라지는 관계로 이해하면 된다.

의존 관계는 한 클래스가 다른 클래스를 일시적으로 사용하는 관계이다. 연관이 비교적 지속적인 연결이라면, 의존은 메서드의 매개변수나 내부 생성처럼 잠깐 사용하는 관계에 가깝다. 실체화 관계는 인터페이스와 구현 클래스 사이의 관계이다. 인터페이스가 “이 기능을 제공해야 한다”는 약속을 제시하면, 구체 클래스가 그 약속을 실제로 구현한다.

확인 포인트:
- 연관은 클래스 사이의 의미 있는 연결이다.
- 일반화는 상속과 관련된 `is-a` 관계이다.
- 집합과 합성은 전체-부분 관계이며 생명주기 독립성으로 구분한다.

4. ①. 실체화는 인터페이스를 구체 클래스가 구현하는 관계이다.
5. ④. 합성은 집합보다 강한 전체-부분 관계이다.

## 9. SOLID 설계 원칙

### 강화 지문

SOLID는 객체지향 설계를 변경에 강하게 만들기 위한 다섯 가지 원칙이다. 이 원칙들은 서로 다른 이름을 가지고 있지만 공통 목표는 비슷하다. 클래스가 너무 많은 일을 떠안지 않게 하고, 새 기능이 추가되어도 기존 코드를 덜 고치게 하며, 구체적인 구현에 강하게 묶이지 않게 하는 것이다. 결국 유지보수성과 확장성을 높이려는 원칙들이다.

SRP, 단일 책임 원칙은 클래스가 변경되는 이유를 하나만 갖도록 설계하라는 원칙이다. 여기서 책임은 단순히 기능 개수 하나라는 뜻보다 “변경 이유”에 가깝다. 한 클래스가 계산도 하고 저장도 하고 화면 출력도 한다면, 계산 방식이 바뀔 때도, 저장 방식이 바뀔 때도, 화면이 바뀔 때도 같은 클래스를 수정해야 한다. 이런 구조는 변경에 취약하다.

OCP, 개방 폐쇄 원칙은 확장에는 열려 있고 수정에는 닫혀 있어야 한다는 원칙이다. 새 기능이 생길 때 기존 코드를 계속 수정해야 한다면 변화에 약하다. 반대로 공통 부분을 추상화하고 새 기능은 하위 클래스나 구현체를 추가하는 방식으로 확장할 수 있다면 기존 코드를 덜 건드릴 수 있다. LSP, 리스코프 교체 원칙은 상위 타입 자리에 하위 타입을 넣어도 정상적으로 동작해야 한다는 원칙이다. 상속 구조가 올바르려면 하위 클래스가 상위 클래스의 의미를 깨뜨리면 안 된다.

DIP, 의존 관계 역전 원칙은 구체 클래스보다 추상화에 의존하라는 원칙이다. 구체 클래스는 자주 바뀔 수 있으므로 인터페이스나 추상 클래스에 의존하면 결합도를 낮출 수 있다. ISP, 인터페이스 분리 원칙은 하나의 큰 인터페이스를 모든 클라이언트가 억지로 공유하게 하지 말고, 필요한 기능만 가진 작은 인터페이스로 나누라는 원칙이다. 사용하지 않는 기능에 의존하지 않게 하는 것이 핵심이다.

확인 포인트:
- SRP는 변경 이유를 하나로 줄인다.
- OCP는 확장에는 열고 기존 수정은 줄인다.
- DIP는 구체 클래스가 아니라 추상화에 의존하게 한다.

### 문제

1. SRP에 대한 설명으로 적절한 것은?
   1. 클래스가 변경되는 이유를 하나만 갖도록 설계한다.
   2. 모든 클라이언트가 하나의 큰 인터페이스를 공유해야 한다.
   3. 구체 클래스에 직접 의존해야 한다.
   4. 객체를 하나만 생성한다.
   5. 프로그램을 실행하지 않고 검토한다.

2. OCP의 핵심 설명으로 적절한 것은?
   1. 확장에는 열려 있고 수정에는 닫혀 있어야 한다.
   2. 하위 클래스는 상위 클래스를 절대 대체할 수 없다.
   3. 내부 구현을 모두 공개해야 한다.
   4. 조건문의 모든 조합을 테스트해야 한다.
   5. 모든 모듈은 높은 결합도를 가져야 한다.

3. LSP에 대한 설명으로 적절한 것은?
   1. 상위 타입 자리에 하위 타입을 넣어도 정상 동작해야 한다.
   2. 입력 값을 경계 중심으로 테스트한다.
   3. 모델, 뷰, 제어를 분리한다.
   4. 주석은 코드와 달라도 된다.
   5. 중앙 리포지토리를 사용한다.

4. DIP의 핵심은?
   1. 구체 클래스보다 추상화에 의존한다.
   2. 모든 객체를 하나의 클래스에서 직접 생성한다.
   3. 테스트를 마지막에만 수행한다.
   4. 모듈 내부 응집도를 낮춘다.
   5. 인터페이스를 하나로 크게 합친다.

5. ISP에 대한 설명으로 적절하지 않은 것은?
   1. 클라이언트가 필요한 기능만 가진 인터페이스로 나눈다.
   2. 불필요한 의존을 줄이는 데 도움이 된다.
   3. 하나의 거대한 인터페이스를 모든 클라이언트가 억지로 공유하게 한다.
   4. 유지보수성 향상과 관련된다.
   5. 인터페이스를 분리하는 원칙이다.

### 정답 및 해설

1. ①. SRP는 변경 이유를 하나로 줄이라는 원칙이다.
2. ①. OCP는 확장에는 열고 기존 수정은 줄이라는 원칙이다.
3. ①. LSP는 하위 타입이 상위 타입을 대체 가능해야 한다는 원칙이다.
4. ①. DIP는 추상화에 의존하여 결합도를 낮춘다.
5. ③. ISP는 큰 인터페이스 강제가 아니라 필요한 인터페이스로 분리하라는 원칙이다.

## 10. 구현과 표준 코딩 규칙

### 강화 지문

구현은 설계 결과를 실제 코드로 옮기는 단계이다. 여기서 중요한 것은 단순히 실행되는 코드를 만드는 것이 아니다. 소프트웨어는 한 사람이 한 번 만들고 끝내는 경우보다, 여러 사람이 함께 만들고 오랫동안 수정하는 경우가 많다. 따라서 구현 단계에서는 다른 사람이 읽고 이해하고 고칠 수 있는 코드가 중요하다. 코드가 돌아가더라도 읽기 어렵고 수정하기 어렵다면 좋은 구현이라고 보기 어렵다.

교안에서는 프로그래밍 언어의 역사도 다룬다. 포트란은 과학 계산 분야와 관련된 초창기 고급 언어이고, 베이직은 쉬운 문법을 가진 교육용 언어로 설명된다. C는 널리 쓰이는 명령형 언어이고, C++은 C에 객체지향 개념을 더한 언어이다. C#은 닷넷 프레임워크와 관련된 객체지향 언어이며, 자바는 JVM을 통해 플랫폼 독립성을 가진다는 점이 중요하다. 이 부분은 세부 문법보다 언어별 대표 특징을 구분하는 것이 핵심이다.

표준 코딩 규칙은 개발자들이 일관된 방식으로 코드를 작성하게 하는 기준이다. 같은 프로젝트에서 사람마다 들여쓰기, 이름 짓기, 주석 방식이 모두 다르면 코드를 읽는 데 불필요한 시간이 든다. 표준 코딩 규칙은 이런 혼란을 줄이고 가독성, 유지보수성, 협업 효율을 높인다. 즉 코딩 규칙은 컴퓨터보다 사람을 위한 규칙에 가깝다.

시험에서는 표준 코딩 규칙의 목적을 실행 속도 향상 하나로만 보는 선지를 조심해야 한다. 물론 깔끔한 코드는 오류를 줄이고 간접적으로 품질을 높일 수 있지만, 핵심은 코드가 읽기 쉽고 일관되며 수정하기 쉬워지는 것이다. 구현 단계의 품질은 나중의 유지보수 비용과 직접 연결된다.

확인 포인트:
- 구현은 설계 결과를 코드로 옮기는 단계이다.
- 자바는 JVM을 통한 플랫폼 독립성이 중요하다.
- 표준 코딩 규칙은 가독성, 유지보수성, 협업 효율을 높인다.

---

   1. 설계 결과를 실제 코드로 옮긴다.
   2. 사용자 요구사항을 처음 수집한다.
   3. 프로그램을 실행하지 않고만 테스트한다.
   4. 객체를 하나만 생성하도록 제한한다.

   3. 교육용으로 쉬운 문법만을 목적으로 만들어졌다.

   3. 테스트 절차의 마지막 단계이다.

   1. 구현에서는 코드의 실행 여부만 중요하고 가독성은 중요하지 않다.
   2. 표준 코딩 규칙은 사람이 코드를 읽고 관리하기 쉽게 한다.
   3. 베이직은 JVM을 통해 플랫폼 독립성을 가진다.
   4. 포트란은 객체를 하나만 생성하는 디자인 패턴이다.
   5. C#은 동등 분할 테스트 기법이다.

### 정답 및 해설

2. ①. 자바는 JVM을 통한 플랫폼 독립성이 중요하다.
3. ①. C++은 C에 객체지향 개념을 더한 언어로 설명된다.
4. ⑤. 코딩 규칙이 테스트를 불필요하게 만들지는 않는다.
5. ②. 표준 코딩 규칙은 가독성, 유지보수성, 협업에 도움을 준다.

## 11. 주요 표준 코딩 규칙

### 강화 지문

표준 코딩 규칙은 코드의 여러 부분을 다룬다. 먼저 명칭 규칙은 변수, 함수, 상수, 매크로 등의 이름을 어떻게 지을지 정한다. 이름은 코드의 의미를 전달하는 가장 빠른 단서이다. 변수명과 함수명이 헷갈리거나 의미 없는 이름을 쓰면 코드를 읽는 사람이 계속 추측해야 한다. 매크로명과 상수명은 대문자와 언더바를 사용하는 식으로 구분할 수 있고, 포인터 변수 앞에 `p`를 붙이면 포인터임을 쉽게 알 수 있다.

소스 형식 규칙은 파일 길이, 한 줄 길이, 함수 길이, 중괄호 위치, 들여쓰기 등을 다룬다. 한 줄이 지나치게 길면 읽기 어렵고, 함수 하나가 너무 길면 그 함수가 정확히 무슨 일을 하는지 파악하기 어렵다. 같은 수준의 문장을 같은 위치에서 시작하도록 들여쓰기를 맞추면 조건문과 반복문의 구조가 눈에 들어온다. 형식은 겉모습의 문제가 아니라 이해 속도와 오류 예방에 연결된다.

주석은 코드의 목적과 사용 방법을 설명한다. 함수 앞에는 목적, 매개변수, 반환 값, 변경 이력 등을 적을 수 있다. 하지만 주석은 많다고 무조건 좋은 것이 아니다. 코드가 바뀌었는데 주석이 그대로 남아 있으면 오히려 잘못된 정보를 준다. 따라서 원시 코드와 주석은 항상 일치하도록 관리해야 한다.

변수와 자료형 규칙도 중요하다. 필요한 변수만 선언하고, 지역 변수는 선언할 때 초기화하는 것이 좋다. 초기화하지 않은 변수는 예측하기 어려운 값을 가질 수 있기 때문이다. 상수는 의미 없는 숫자를 코드에 직접 쓰기보다 이름 있는 상수로 표현하면 의미가 분명해진다. 수식에서는 연산자 우선순위를 명확히 하기 위해 괄호를 사용할 수 있고, `goto` 문은 흐름을 복잡하게 하므로 사용을 자제한다. `switch` 문에서는 각 `case`를 빠져나갈 때 보통 `break`가 필요하다.

확인 포인트:
- 명칭 규칙은 이름만 보고 역할을 이해하게 돕는다.
- 주석은 코드와 일치하도록 관리해야 한다.
- `goto`는 흐름을 복잡하게 할 수 있어 사용을 자제한다.

   2. 모든 변수를 같은 이름으로 만드는 것
   3. 테스트 케이스를 줄이는 것

   5. 내부 구현을 외부에 공개하는 것

   1. 연산자가 여러 개이면 괄호를 사용해 우선순위를 명확히 할 수 있다.
   2. 증감 연산자는 다른 연산자와 최대한 복잡하게 섞어야 한다.
   3. 상수는 반드시 의미 없는 숫자로 직접 작성해야 한다.
   4. 들여쓰기는 코드 이해와 무관하다.
   5. 포인터 변수명은 항상 함수명과 같아야 한다.

### 정답 및 해설

1. ①. 명칭 규칙은 이름으로 역할을 쉽게 파악하게 한다.
2. ②. 주석은 코드 변경에 맞게 함께 관리되어야 한다.
3. ①. 지역 변수는 선언 시 초기화하는 것이 좋다.
4. ⑤. `switch`의 `case` 종료에는 보통 `break`가 필요하다.
5. ①. 괄호는 수식의 우선순위를 명확히 하는 데 도움을 준다.

## 12. 디자인 패턴의 분류와 행위 패턴

### 강화 지문

디자인 패턴은 반복적으로 발생하는 설계 문제에 대한 재사용 가능한 해결책이다. 이것은 특정 언어의 문법이나 복사해서 붙여 넣는 코드 조각이 아니다. 여러 프로젝트에서 비슷한 설계 문제가 반복되기 때문에, 이미 검증된 해결 방식을 정리해 둔 설계 지식이다. 패턴을 알면 매번 처음부터 시행착오를 겪지 않고, 문제 상황에 맞는 설계 방향을 빠르게 잡을 수 있다.

디자인 패턴은 크게 생성 패턴, 구조 패턴, 행위 패턴으로 나뉜다. 생성 패턴은 객체를 어떻게 만들 것인가를 다룬다. 객체 생성 책임이 특정 클래스에 너무 강하게 묶이면 새 객체 종류가 생길 때 기존 코드가 계속 수정될 수 있다. 구조 패턴은 클래스나 객체를 어떻게 조합할 것인가를 다룬다. 여러 객체를 조합하여 더 큰 구조를 만들거나, 서로 맞지 않는 인터페이스를 연결하는 문제가 여기에 속한다.

행위 패턴은 객체들이 어떻게 상호작용하고 책임을 나눌 것인가를 다룬다. 객체 하나가 모든 일을 처리하는 구조는 변경에 취약하다. 여러 객체가 각자의 역할을 맡고 메시지를 주고받는 구조가 필요하다. 하지만 이 상호작용이 복잡해지면 객체들이 서로 강하게 의존할 수 있다. 행위 패턴은 이런 책임 분배와 메시지 교환의 문제를 해결한다.

교안에서 다루는 대표적인 행위 패턴은 Strategy와 State이다. Strategy는 자주 바뀌는 기능이나 알고리즘을 별도 클래스로 분리한다. State는 객체의 상태가 자주 바뀌고 상태에 따라 행동이 달라질 때 상태를 클래스로 분리한다. 둘 다 변화하는 부분을 찾아 분리한다는 공통점이 있지만, Strategy는 기능 교체에 가깝고 State는 상태 변화에 가깝다.

확인 포인트:
- 디자인 패턴은 반복되는 설계 문제의 재사용 가능한 해결책이다.
- 생성 패턴은 객체 생성, 구조 패턴은 객체 조합, 행위 패턴은 상호작용을 다룬다.
- Strategy는 기능 교체, State는 상태 변화에 초점을 둔다.

4. ①. 행위 패턴은 상호작용과 책임 분배를 다룬다.
5. ①. Strategy는 기능 교체, State는 상태 변화가 핵심이다.

## 13. Strategy와 State 패턴

### 강화 지문

Strategy 패턴은 기능이나 알고리즘이 자주 바뀔 때 유용하다. 예를 들어 포켓몬 게임에서 포켓몬마다 공격 기술과 패시브 기술이 있다고 하자. 처음에는 몇 가지 기술만 있더라도 시간이 지나면 새로운 기술이 추가되고 기존 기술이 바뀔 수 있다. 이 기능을 포켓몬 클래스 내부 메서드로 고정해 두면 기술이 바뀔 때마다 포켓몬 클래스가 계속 수정된다. 이렇게 변화가 잦은 부분이 주 클래스 안에 박혀 있으면 구조가 점점 복잡해진다.

Strategy 패턴은 먼저 “자주 바뀌는 것이 무엇인가”를 찾는다. 그다음 그 기능을 별도의 전략 클래스로 분리하고, 공통 인터페이스를 통해 교체할 수 있게 한다. 포켓몬 클래스는 구체적인 공격 기술 클래스를 직접 알아야 하는 것이 아니라, 공격 인터페이스를 통해 기능을 사용한다. 새로운 공격 기술이 생기면 새 전략 클래스를 추가하면 되므로 기존 구조를 덜 수정할 수 있다. 이 점에서 OCP와도 연결된다.

State 패턴은 객체의 상태가 자주 바뀌고, 상태에 따라 행동이 달라질 때 사용한다. 선풍기를 예로 들면 정지, 송풍, 약풍, 강풍 같은 상태가 있고 버튼을 누를 때마다 동작이 달라진다. 이 상태들을 하나의 클래스 안에서 `if ~ else if`로 모두 처리하면 상태가 추가될수록 조건문이 길어진다. 상태 변경 규칙이 복잡해질수록 주 클래스가 계속 수정되고 읽기 어려워진다.

State 패턴은 상태 자체를 클래스로 만든다. 주 클래스는 현재 상태 객체를 가지고 있고, 버튼 입력 같은 요청이 들어오면 현재 상태 객체에게 행동을 위임한다. 상태의 추가나 변경은 상태 클래스 쪽에서 이루어지므로 주 클래스에 미치는 영향을 줄일 수 있다. 시험에서는 Strategy와 State를 구분하는 것이 중요하다. 기능이나 알고리즘이 바뀌면 Strategy, 상태와 상태 전이가 바뀌면 State로 생각하면 된다.

확인 포인트:
- Strategy는 자주 바뀌는 기능이나 알고리즘을 분리한다.
- State는 자주 바뀌는 상태와 상태별 행동을 분리한다.
- 두 패턴 모두 변화하는 부분을 찾아 주 클래스의 수정을 줄인다.

4. ④. 객체 하나만 생성은 Singleton의 핵심이다.

## 14. Decorator와 Adapter 패턴

### 강화 지문

Decorator 패턴은 객체에 부가 기능을 동적으로 추가할 때 사용한다. 토스트 예시를 생각하면 기본 식빵은 핵심 객체이고 치즈, 햄, 달걀 같은 토핑은 부가 기능이다. 만약 모든 토핑 조합을 상속으로만 만들면 클래스 수가 빠르게 늘어난다. 치즈토스트, 햄토스트, 치즈햄토스트, 치즈햄달걀토스트처럼 조합마다 클래스를 만들면 유지보수가 어렵다.

Decorator 패턴은 기본 객체를 감싸는 장식 객체를 사용한다. 장식 객체는 원래 객체와 같은 타입처럼 행동하면서 추가 기능을 덧붙인다. 이렇게 하면 실행 중에도 필요한 장식을 조합할 수 있다. 핵심은 원래 객체의 구조를 크게 바꾸지 않고 부가 기능을 붙일 수 있다는 점이다. 이 패턴은 상속보다 구성과 위임을 활용해 유연한 확장을 가능하게 한다.

Adapter 패턴은 서로 호환되지 않는 인터페이스를 함께 사용할 수 있게 해 준다. 어떤 시스템은 `play()`와 `stop()`이라는 메서드를 기대하는데, 외부 객체는 `playing()`과 `stopping()`이라는 메서드를 제공한다고 하자. 기존 시스템을 모두 고치거나 외부 객체를 직접 수정하기 어렵다면 중간에 어댑터를 둔다. 어댑터는 기존 시스템이 기대하는 인터페이스를 제공하고, 내부적으로 외부 객체의 실제 메서드를 호출한다.

Decorator와 Adapter는 모두 구조 패턴이지만 목적이 다르다. Decorator는 같은 인터페이스를 유지하면서 부가 기능을 붙이는 것이고, Adapter는 맞지 않는 인터페이스를 변환해 연결하는 것이다. 시험에서는 이 둘이 자주 섞인다. “부가 기능을 동적으로 추가한다”는 Decorator, “호환되지 않는 인터페이스를 연결한다”는 Adapter로 잡으면 된다.

확인 포인트:
- Decorator는 객체에 부가 기능을 동적으로 추가한다.
- Adapter는 호환되지 않는 인터페이스를 연결한다.
- Decorator는 기능 추가, Adapter는 인터페이스 변환이 핵심이다.

   2. Adapter는 인터페이스 차이를 맞추는 데 유용하다.
   3. Decorator와 Adapter는 모두 구조 패턴으로 볼 수 있다.
   4. Adapter는 객체를 오직 하나만 생성하도록 보장하는 패턴이다.
   5. Decorator는 클래스 조합 폭발을 줄이는 데 도움이 될 수 있다.

### 정답 및 해설

1. ①. Decorator는 부가 기능을 동적으로 덧붙인다.
2. ①. Adapter는 호환되지 않는 인터페이스를 연결한다.
3. ①. 토핑처럼 기능을 붙이는 예는 Decorator에 가깝다.
4. ①. 인터페이스 변환자는 Adapter의 핵심이다.
5. ④. 객체 하나만 생성은 Singleton의 핵심이다.

## 15. Factory Method와 Singleton 패턴

### 강화 지문

Factory Method 패턴은 객체 생성 책임을 분리하는 패턴이다. 한 클래스가 여러 구체 클래스를 직접 `new`로 생성하면, 새 종류가 추가될 때마다 그 클래스가 수정된다. 예를 들어 게임 서버가 슈퍼마리오, 테트리스 같은 구체 게임 객체를 직접 생성한다고 하자. 새 게임이 추가될 때마다 게임 서버 코드를 고쳐야 한다면 OCP 관점에서 좋지 않다. 확장은 새 게임 추가로 이루어져야 하는데 기존 코드 수정이 계속 필요하기 때문이다.

Factory Method 패턴은 객체를 생성하는 책임을 별도의 메서드나 클래스 쪽으로 옮긴다. 클라이언트는 구체 객체를 직접 만들기보다 생성 메서드를 통해 객체를 얻는다. 이렇게 하면 생성 방식이 바뀌거나 새로운 객체 종류가 추가되어도 기존 코드의 수정 부담을 줄일 수 있다. 또한 구체 클래스에 직접 의존하기보다 추상 타입을 통해 다루면 결합도도 낮아진다.

Singleton 패턴은 객체가 프로그램 안에서 오직 하나만 생성되도록 보장하는 패턴이다. 로그 기록 객체, 캐시, 설정 관리자처럼 여러 개가 따로 생기면 문제가 될 수 있는 객체에 사용한다. 핵심은 외부에서 마음대로 객체를 만들지 못하게 하고, 정해진 하나의 객체를 공유하게 하는 것이다.

일반적인 Singleton 구현에서는 생성자를 `private`으로 두어 외부 생성을 막는다. 클래스 내부에 `static` 객체를 두고, 외부에서는 `getInstance()` 같은 메서드로 그 객체를 얻는다. 스레드가 여러 개 동시에 실행되는 환경에서는 두 스레드가 동시에 객체를 만들지 않도록 동기화 문제도 고려해야 한다. Factory Method가 “누가 객체를 만들 것인가”의 문제라면, Singleton은 “객체를 몇 개 만들 것인가”의 문제이다.

확인 포인트:
- Factory Method는 객체 생성 책임을 분리한다.
- Singleton은 객체가 하나만 생성되도록 보장한다.
- Singleton은 private 생성자, static 객체, getInstance와 연결된다.

## 16. 테스트의 필요성, 용어, 절차

### 강화 지문

소프트웨어 테스트는 프로그램에 숨어 있는 오류를 발견하고, 시스템이 요구사항을 만족하는지 확인하는 활동이다. 테스트를 단순히 “마지막에 한번 실행해 보는 것”으로 생각하면 부족하다. 소프트웨어 오류는 작은 불편을 넘어서 큰 사고로 이어질 수도 있다. 그래서 테스트는 문제를 사후에 처리하는 일이 아니라, 결함을 조기에 발견하고 신뢰를 높이는 과정이다.

테스트의 좁은 의미는 원시 코드 속 오류를 찾는 것이다. 그러나 넓은 의미에서는 소프트웨어가 고객이 원한 기능을 제대로 제공하는지, 실제로 사용할 만한 품질을 갖추었는지를 확인하는 것이다. 개발자는 자신이 만든 코드가 맞다고 생각할 수 있지만, 실제 사용자의 요구와 다르거나 특정 상황에서 잘못 동작할 수 있다. 테스트는 이런 차이를 드러내는 역할을 한다.

오류, 결함, 고장은 구분해서 알아야 한다. 오류는 개발자가 저지른 실수이다. 예를 들어 조건식을 잘못 작성한 것이 오류이다. 그 오류가 코드나 문서 안에 남아 있으면 결함이 된다. 그리고 그 결함이 실행 중에 잘못된 결과나 동작으로 드러나면 고장이라고 할 수 있다. 즉 오류는 원인에 가깝고, 결함은 내부에 남은 문제이며, 고장은 외부로 나타난 현상이다.

테스트는 절차를 가진다. 먼저 테스트 계획 단계에서 테스트 목표, 대상, 범위, 계획서를 정한다. 다음으로 테스트 설계 단계에서 적절한 테스트 기법을 고르고 테스트 케이스와 테스트 데이터를 만든다. 테스트 실행 및 측정 단계에서는 환경을 구축하고 실제 테스트를 수행하며 결과를 기록한다. 마지막으로 오류 추적 및 수정 단계에서는 오류 위치와 원인을 확인하고 수정한 뒤 다시 확인한다. 시험에서 순서 문제가 나오면 이 흐름을 기억하면 된다.

확인 포인트:
- 테스트는 오류 발견과 요구사항 만족 여부 확인을 위한 활동이다.
- 오류는 개발자의 실수, 결함은 내부 문제, 고장은 외부로 드러난 잘못된 동작이다.
- 테스트 절차는 계획, 설계, 실행 및 측정, 오류 추적 및 수정의 흐름이다.

3. ①. 테스트 계획은 목표와 대상, 범위를 정한다.
4. ①. 테스트 설계에서는 기법과 테스트 케이스를 준비한다.
5. ①. 테스트 절차는 계획, 설계, 실행, 추적 및 수정 흐름이다.

## 17. 정적 테스트, 동적 테스트, 블랙박스 기법

### 강화 지문

정적 테스트와 동적 테스트는 프로그램을 실행하는지 여부로 구분한다. 정적 테스트는 프로그램을 실행하지 않고 문서, 설계서, 코드 같은 산출물을 검토하여 결함을 찾는다. 자동차를 실제로 운전하지 않고 보닛을 열어 부품 상태를 확인하는 것과 비슷하다. 실행 전에도 요구사항이 제대로 반영되었는지, 표준을 지켰는지, 코드나 문서에 오류가 없는지 확인할 수 있다.

정적 테스트에는 개별 검토, 동료 검토, 검토회의, 소프트웨어 검사 등이 있다. 개별 검토는 본인이 체크리스트로 자신의 산출물을 확인하는 방식이다. 동료 검토는 다른 사람이 검토해 주는 방식이고, 검토회의는 여러 전문가가 절차에 따라 산출물을 검토하는 방식이다. 소프트웨어 검사는 공식 검토에 속하며, 코드뿐 아니라 각 단계의 문서까지 분석하고 품질을 평가한다. 검사 회의의 목적은 오류를 발견하는 것이지 그 자리에서 모든 오류를 수정하는 것이 아니다.

동적 테스트는 프로그램을 실제로 실행하여 입력에 대한 출력이나 실행 상태를 확인한다. 그중 명세 기반 테스트, 즉 블랙박스 테스트는 내부 구조를 보지 않고 요구분석명세서와 기능 명세를 기준으로 한다. 사용자가 어떤 입력을 넣었을 때 기대한 출력이 나오는지를 확인하는 방식이다. 내부 알고리즘을 보지 않기 때문에 사용자의 관점에서 기능을 확인하기 좋다.

블랙박스 기법에는 신텍스 기법, 동등 분할, 경계 값 분석, 원인-결과 그래프가 있다. 신텍스 기법은 문법에 맞는 입력과 맞지 않는 입력을 확인한다. 동등 분할은 입력 영역을 같은 성격의 그룹으로 나누어 대표 값을 테스트한다. 경계 값 분석은 오류가 자주 발생하는 경계와 그 주변 값을 테스트한다. 원인-결과 그래프는 여러 입력 조건과 결과의 관계를 그래프로 나타내고, 이를 의사결정 테이블로 바꾸어 테스트 케이스를 만든다.

확인 포인트:
- 정적 테스트는 실행하지 않고 검토한다.
- 블랙박스 테스트는 내부 구조를 보지 않고 명세와 입출력을 본다.
- 동등 분할은 대표 값, 경계 값 분석은 경계 주변 값, 원인-결과 그래프는 조건 조합을 다룬다.

   4. 상태를 클래스로 분리한다.
   5. 사용자가 인수 전 테스트한다.

5. 원인-결과 그래프 기법과 관련 깊은 것은?
   1. 의사결정 테이블
   2. 생성자 private
   3. MVC
   4. 높은 응집도
   5. 4+1 관점

### 정답 및 해설

1. ①. 정적 테스트는 실행하지 않고 검토한다.
2. ⑤. 검사 회의는 오류 발견이 목적이며 그 자리에서 수정하는 것이 핵심이 아니다.
3. ①. 블랙박스는 내부를 보지 않고 명세와 입출력을 본다.
4. ①. 동등 분할은 입력 영역을 동등 클래스로 나누어 대표 값을 고른다.
5. ①. 원인-결과 그래프는 의사결정 테이블로 변환된다.

## 18. 화이트박스, V 모델, 단계별 테스트와 회귀 테스트

### 강화 지문

화이트박스 테스트는 프로그램 내부 구조를 보면서 테스트하는 구현 기반 테스트이다. 블랙박스 테스트가 내부를 보지 않고 명세와 입출력에 집중한다면, 화이트박스 테스트는 코드 내부의 조건문, 경로, 변수, 제어 흐름을 본다. 따라서 개발자 관점에서 코드가 어떤 경로를 지나가는지 확인하는 데 유용하다.

화이트박스 테스트에는 여러 검증 기준이 있다. 문장 검증 기준은 모든 문장이 최소 한 번 실행되도록 테스트한다. 하지만 모든 문장을 지나갔다고 해서 조건의 참과 거짓이 모두 확인되는 것은 아니다. 분기 검증 기준은 조건문의 참과 거짓 분기가 최소 한 번씩 실행되도록 한다. 조건 검증 기준은 복합 조건문 안의 개별 조건식이 참과 거짓을 모두 갖도록 한다. 다중 조건 검증 기준은 조건 조합을 더 충분히 다루어 가려지는 오류를 줄이려 한다.

기본 경로 테스트는 원시 코드의 독립적인 경로가 최소 한 번은 실행되도록 테스트 케이스를 찾는 방법이다. 이때 순환 복잡도는 코드의 복잡도를 수치로 나타내며 독립 경로 수와 연결된다. 공식은 `CC = R`, `CC = E - N + 2`, `CC = P + 1`로 정리할 수 있다. 순서 문제로 나오면 순서도 작성, 흐름 그래프 작성, 순환 복잡도 계산, 독립 경로 정의, 테스트 케이스 작성의 흐름을 기억하면 된다.

V 모델은 개발 단계와 테스트 단계를 대응시켜 보여 준다. 작은 단위에서 큰 단위로 보면 단위 테스트, 통합 테스트, 시스템 테스트, 인수 테스트이다. 단위 테스트는 모듈 하나를 확인하고, 통합 테스트는 모듈 사이의 상호작용을 확인한다. 하향식 통합에는 아직 없는 하위 모듈을 대신하는 스텁이 필요할 수 있고, 상향식 통합에는 상위 모듈 역할을 하는 드라이버가 필요할 수 있다. 시스템 테스트는 전체 시스템을 확인하고, 인수 테스트는 사용자가 인수 전 요구사항 충족 여부를 확인한다. 확정 테스트는 수정한 결함이 제대로 고쳐졌는지 확인하고, 회귀 테스트는 수정이 다른 부분에 영향을 주었는지 확인한다.

확인 포인트:
- 화이트박스 테스트는 코드 내부 구조와 경로를 본다.
- 순환 복잡도 공식은 `CC = R`, `CC = E - N + 2`, `CC = P + 1`이다.
- 회귀 테스트는 수정이 다른 부분에 미친 영향을 확인한다.

### 문제

1. 화이트박스 테스트에 대한 설명으로 적절한 것은?
   1. 코드 내부 구조와 실행 경로를 분석한다.
   2. 내부 구조를 보지 않고 명세만 본다.
   3. 객체를 하나만 생성한다.
   4. 사용자 화면의 일관성을 설계한다.
   5. 중앙 리포지토리에 데이터를 저장한다.

2. 문장 검증 기준에 대한 설명으로 적절한 것은?
   1. 모든 문장이 최소 한 번 실행되도록 한다.
   2. 모든 사용자 요구사항을 처음 수집한다.
   3. 객체 생성 책임을 분리한다.
   4. 인터페이스를 변환한다.
   5. 주석을 모두 삭제한다.

3. 순환 복잡도 공식으로 적절하지 않은 것은?
   1. CC = R
   2. CC = E - N + 2
   3. CC = P + 1
   4. CC = 응집도 - 결합도
   5. 간선과 노드 수를 이용해 계산할 수 있다.

4. 개발 단계별 테스트를 작은 단위에서 큰 단위로 배열한 것은?
   1. 단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트
   2. 인수 테스트 → 시스템 테스트 → 통합 테스트 → 단위 테스트
   3. 통합 테스트 → 단위 테스트 → 인수 테스트 → 시스템 테스트
   4. 회귀 테스트 → 확정 테스트 → 설계 → 구현
   5. 블랙박스 → 코딩 규칙 → 아키텍처 → UI 설계

5. 회귀 테스트에 대한 설명으로 적절한 것은?
   1. 수정이 다른 부분에 영향을 주었는지 확인한다.
   2. 객체를 하나만 생성했는지 확인한다.
   3. 내부 구현을 외부에 공개한다.
   4. 입력 값의 문법만 검사한다.
   5. 모델, 뷰, 제어를 분리한다.

### 정답 및 해설

1. ①. 화이트박스는 내부 구조와 경로를 본다.
2. ①. 문장 검증은 모든 문장을 최소 한 번 실행하게 한다.
3. ④. 응집도와 결합도의 차이는 순환 복잡도 공식이 아니다.
4. ①. 작은 단위에서 큰 단위로 단위, 통합, 시스템, 인수 테스트이다.
5. ①. 회귀 테스트는 수정이 다른 기능에 미친 영향을 확인한다.

고객의 요구사항을 반영하여 설계 도면을 만들고, 이를 바탕으로 건물을 짓는다.

| 단계 | 건축 설계 과정 설명 |
|---|---|
| 1. 요구사항 수집 | 건축주는 짓고 싶은 집의 모양을 구상하고 외관, 방의 개수 및 위치 등을 설계사에게 제시한다. |
| 2. 설계 도면 작성 | 설계사는 건축주의 요구사항과 건축 경험을 바탕으로 고려해야 할 제약 사항을 반영하여 도면을 작성한다. |
| 3. 설계 확정 및 시공 | 건축주와 설계사가 의견을 조율하여 최종 설계 도면을 확정한 후 시공을 진행한다. |

**답:** 설계 도면

## 7페이지 - 요구분석과 설계의 차이

### 원문 전사

**상단 제목:** 설계의 이해  
**본문 제목:** 요구 분석과 설계의 차이  
**표 5-1:** 요구분석과 설계의 차이

| 구분 | 요구분석 | 설계 |
|---|---|---|
| 산출물 | 요구분석명세서 | 설계서 |
| 관점 | 무엇(what)을 만들 것인가 | 어떻게(how) 만들 것인가 |
| 특성 | 개념적, 추상적 | 사용 환경을 반영해 구체적 |
| 비고 | 요구분석 단계에서 고려하지 않는 요소: 운영체제, DBMS(오라클, MySQL, MS-SQL Server 등), 프레임워크(.NET, J2EE 등) | 설계 단계에서 고려하는 요소: 비기능 요구사항, 제약 사항, 플랫폼(운영체제, 미들웨어, 프레임워크) |

설계 단계에서는 다음과 같은 플랫폼과 구현 환경을 함께 고려한다.

- OS: Windows, Linux
- Java platform: Java EE framework
- M/S platform: .NET framework
- application server: .NET Framework, BEA WebLogic, IBM WebSphere 등
- DBMS: Oracle, SQL Server 등
- Programming Language: C++, Java, C# 등 선택

### 이해 설명

요구분석은 사용자가 필요로 하는 기능과 조건을 정의한다. 설계는 그 요구를 실제 환경에서 구현할 수 있도록 구체적인 구조와 기술 선택으로 변환한다. 시험에서는 `what`과 `how`, 요구분석명세서와 설계서의 구분이 중요하다.

## 8페이지 - 소프트웨어 설계의 정의

### 원문 전사

**상단 제목:** 설계의 이해  
**본문 제목:** 소프트웨어 설계  
**소제목:** 정의

요구분석명세서를 기반으로 소프트웨어를 어떻게 구축할 것인가를 결정하는 과정.

다양한 방법 중에서 제약 조건을 만족하는 최적의 설계를 도출하는 것이 중요하며, 이를 평가할 수 있는 기준도 정량적으로 명시해야 한다.

### 이해 설명

설계는 단순히 구조를 그리는 행위가 아니다. 여러 가능한 해결책 가운데 요구사항과 제약 조건을 만족하는 해결책을 선택하고, 그 설계가 좋은지 평가할 기준까지 정하는 과정이다.

## 9페이지 - 좋은 설계의 조건

### 원문 전사

**상단 제목:** 설계의 이해  
**본문 제목:** 좋은 설계의 조건

- 요구분석명세서의 내용을 설계서에 모두 포함해야 한다.
- 유지보수가 용이하도록 추적이 가능해야 한다.
- 변화에 쉽게 적응할 수 있어야 한다.
- 시스템 변경으로 인한 영향이 최소화되도록 국지적이어야 한다.
- 설계서는 읽기 쉽고, 이해하기 쉽게 작성해야 한다.

좋은 설계가 되려면:

- module의 독립성
- strongly cohesion
- loosely coupling
- 3절에서 다룸

### 이해 설명

좋은 설계는 요구사항을 빠짐없이 반영하고, 요구사항과 설계 요소 사이를 추적할 수 있어야 한다. 변경이 발생했을 때 전체 시스템으로 영향이 퍼지지 않고 특정 부분에 국한되어야 한다. 모듈 내부는 강한 응집도를, 모듈 사이는 느슨한 결합도를 갖는 것이 바람직하다.

## 12페이지 - 분할과 정복

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 분할과 정복

**분할과 정복(divide & conquer)**

- 큰 문제를 소단위로 나누고, 소단위의 작업을 하나씩 처리하여 전체 일을 끝내는 방법
- 가장 세분화된 작은 시스템을 개발하고, 위로 올라가면서 완성시키는 방법으로 개발

**예: 대학 종합정보시스템**

- A대학교 시스템
  - 학사관리
    - 수업관리
    - 수강관리
    - 학적관리
  - 회계관리

**그림 5-2:** 분할의 예

### 이해 설명

복잡한 시스템을 한 번에 구현하지 않고 관리 가능한 하위 문제로 나눈다. 하위 시스템을 개발한 뒤 조합해 전체 시스템을 완성한다.

## 13페이지 - 적절한 분할 수준

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 분할과 정복

모듈의 크기가 작으면:

- 모듈 개수 많음
- 모듈 간의 통신 횟수 많음
- 복잡도 증가
- 효율성 감소

모듈의 크기가 크면:

- 모듈 개수 적음
- 분할 효과 적음
- 분할 의미 퇴색

**설계자의 역할?**

- 모듈 크기를 고려한 분할 수준 결정

### 이해 설명

모듈을 무조건 작게 나누는 것도, 크게 유지하는 것도 좋지 않다. 너무 작으면 모듈 사이의 통신과 관리 비용이 증가하며, 너무 크면 분할의 장점이 사라진다. 설계자는 적절한 균형점을 정해야 한다.

## 14페이지 - 추상화의 정의

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 추상화

**추상화(abstraction)**

복잡한 정보를 단순화하여 필요한 특징만 드러내고, 불필요한 세부 사항은 숨기는 과정.

### 그림 설명

교통 안내 화면의 예시를 사용해 사용자가 목적지까지 이동하는 데 필요한 거리와 시간 등의 핵심 정보만 보여주는 모습을 나타낸다.

### 이해 설명

추상화는 모든 정보를 제거하는 것이 아니라 현재 목적에 필요한 정보만 강조하는 것이다.

## 22페이지 - 캡슐화의 장점

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 캡슐화  
**소제목:** 캡슐화의 장점

- 데이터 보호: 사용자의 데이터 직접 접근 불가 때문
- 제공자와 이용자를 명확히 분리
- 이용자에게 편리성 제공: method의 구현 방법이 바뀌어도 사용하는 데 불편함 없음
- 사용법이 쉬움: 세탁기의 조작 버튼만 알면 되듯이 method의 기능만 알면 됨
- 변화에 대한 국지적 영향: 내부 데이터 구조가 바뀌어도 다른 객체에는 영향을 주지 않음
- 객체 간의 독립성 보장: 캡슐화로 인해 객체 사이의 독립성이 구조적으로 보장
- 변경 용이성과 재사용성 증대

그림 속 영문:

- ENCAPSULATION
- Variables
- Methods
- Class

`figure` 클래스:

- `-line_color`
- `-hight`
- `+fill_color()`
- `+erase()`
- `+draw()`
- `+area()`

### 이해 설명

캡슐화는 내부 구현 변경이 외부에 미치는 영향을 줄이고, 데이터 보호와 객체 독립성을 높인다.

## 32페이지 - 정보 은닉 표기 및 의미

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 정보 은닉  
**소제목:** 정보 은닉 표기 및 의미  
**표 5-2:** UML, C++, 자바 언어의 정보 은닉 표기

`figure` 클래스:

- `#line_color`
- `-hight`
- `+fill_color()`
- `+erase()`
- `+draw()`
- `+area()`

| 공개 범위 | UML | C++ | 자바 |
|---|---|---|---|
| 공개 | `+` | `public` | `public` |
| 은닉 | `-` | `private` | `private` |
| 부분 공개 | `#` | `protected` | `protected` |

| 설명 | 설정 |
|---|---|
| `+` 공개, public | 같은 시스템에 있는 모든 class가 접근 가능 |
| `-` 은닉, private | 같은 시스템 내의 다른 class는 접근 불가능. 해당 class의 method를 통해서만 접근 가능 |
| `#` 부분 공개, protected | 다른 class는 접근 불가능. 해당 class의 method와 class를 상속받은 하위 class만 접근 가능 |

### 이해 설명

UML 클래스 다이어그램에서는 `+`, `-`, `#` 기호로 가시성을 표현한다. 각각 public, private, protected에 대응한다.

## 33페이지 - 정보 은닉의 장점

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 정보 은닉  
**소제목:** 정보 은닉의 장점

- black-box 역할: data와 method를 숨김, 객체 사용자와 제공자 역할 분리
- interface를 통한 접근: 모듈 구현 형태를 몰라도 interface를 통한 사용법만 알면 이용 가능
- 자료구조 변경 용이: 객체 내부 자료 구조를 변경해도 사용자에게는 영향을 주지 않음
- 독립성 향상: 다른 모듈과의 관계가 적기 때문
- 수정 용이: interface만 바뀌지 않으면 내부 구조 변경 시 사용자에게 영향을 주지 않음
- 이해도 증진: 모듈 구현 내용 알 필요 없이 제공하는 기능만 알면 사용 가능
- 확장성 증가: 모듈 내의 변경이 쉬워 기능 추가 하기도 쉬움

### 이해 설명

정보 은닉은 모듈을 블랙박스로 만든다. 공개 인터페이스가 유지되는 한 내부 구현을 자유롭게 변경할 수 있어 수정, 확장, 재사용이 쉬워진다.

## 40페이지 - method overriding과 LSP

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 재정의

**재정의(再定義, redefinition)**

- 다시(再) 정의(定義), 덮어쓰기
- 앞에서 정의한 것을 다 무시하고 내가 새로 정의해서 사용하겠다는 것

**method overriding**

- 상위 class에서 정의한 method는 무시
- 하위 class에서 새로 정의해 사용

**method overriding을 잘못 사용하면**

- 설계 원칙 위반
- 리스코프 교체 원칙(Liskov Substitution Principle) 위배

### 이해 설명

오버라이딩은 다형성을 구현하는 중요한 방법이지만, 하위 클래스가 상위 클래스의 약속을 깨는 방식으로 동작하면 리스코프 교체 원칙을 위반한다.

## 41페이지 - 재정의 코드 예

### 원문 전사

**상단 제목:** 설계의 원리  
**본문 제목:** 재정의

```java
public interface Vehicle {
    public abstract void start();
    void moveForward(); // public abstract 키워드 생략 가능
    void moveBackward();
}
```

```java
public class Car implements Vehicle { // 이동수단을 구체화한 자동차 클래스
    @Override
    public void moveForward() {
        System.out.println("🚗가 앞으로 전진합니다⏩");
    }

    @Override
    public void moveBackward() {
        System.out.println("🚗가 뒤로 후진합니다⏪");
    }
}
```

### 이해 설명

`Vehicle` 인터페이스가 이동 수단의 공통 동작을 선언하고, `Car`가 이를 구현하면서 자동차에 맞는 동작으로 재정의한다.

# 전체 개념 연결 정리

## 1. 요구분석과 설계

요구분석은 무엇을 만들지 정의하고 요구분석명세서를 만든다. 설계는 이를 실제 환경에서 어떻게 만들지 구체화해 설계서를 만든다. 설계에서는 비기능 요구사항, 제약 사항, 운영체제, 미들웨어, 프레임워크, DBMS, 프로그래밍 언어 등을 고려한다.

## 2. 좋은 설계의 방향

좋은 설계는 요구사항을 빠짐없이 포함하고 추적 가능해야 한다. 변화에 쉽게 적응하며, 변경의 영향은 국지적이어야 한다. 문서는 읽고 이해하기 쉬워야 하며, 모듈은 독립적이고 강한 응집도와 느슨한 결합도를 지향한다.

## 3. 설계 원리

- 분할과 정복: 큰 문제를 적절한 크기의 작은 문제로 나눈다.
- 추상화: 현재 목적에 필요한 핵심 정보만 추출하고 불필요한 세부를 생략한다.
- 캡슐화: 관련 데이터와 메서드를 클래스에 묶고 경계를 만든다.
- 정보 은닉: 내부 데이터와 구현을 감추고 인터페이스를 통해서만 접근하게 한다.
- 다형성: 같은 이름이나 인터페이스가 상황과 객체에 따라 다른 동작을 수행한다.

## 4. 중복정의와 재정의

| 구분 | 중복정의 | 재정의 |
|---|---|---|
| 영문 | overloading | overriding |
| 위치 | 한 클래스 안 | 상속 또는 인터페이스 구현 관계 |
| 구별 기준 | 매개변수 개수 또는 자료형 | 상위 메서드와 같은 호출 계약 |
| 목적 | 같은 이름으로 다양한 입력 처리 | 하위 클래스에 맞는 동작 구현 |

## 5. 정보 은닉 가시성

| UML | 언어 키워드 | 의미 |
|---|---|---|
| `+` | `public` | 모든 외부 객체에 공개 |
| `-` | `private` | 해당 클래스 내부에만 공개 |
| `#` | `protected` | 해당 클래스와 하위 클래스에 공개 |

## 6. 시험 핵심 문장

- 요구분석은 `what`, 설계는 `how`이다.
- 설계는 요구분석명세서를 기반으로 소프트웨어를 어떻게 구축할지 결정하는 과정이다.
- 좋은 설계는 변경의 영향이 국지적이고 추적 가능하며 이해하기 쉬워야 한다.
- 분할은 너무 작거나 너무 커도 좋지 않으며 적절한 분할 수준을 정해야 한다.
- 추상화는 필수 정보를 강조하고 관련 없는 세부 사항을 생략한다.
- 캡슐화만으로 정보가 자동으로 숨겨지는 것은 아니며 정보 은닉이 필요하다.
- 객체 간 메시지는 공개 인터페이스인 메서드를 통해 전달한다.
- public은 공개, private은 은닉, protected는 부분 공개이다.
- 오버로딩은 매개변수의 개수 또는 자료형을 다르게 한다.
- 오버라이딩을 잘못 사용하면 리스코프 교체 원칙을 위반할 수 있다.
